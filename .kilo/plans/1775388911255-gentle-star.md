## Priority 1: CRITICAL Issues (Must Fix)

### 1.1 Publishing Feature Completely Non-Functional
- **File:** `server/app/routes/publish.py:24-63`
- **Root Cause:** `_execute_publish()` is defined but never called. `create_publish_tasks` creates tasks with `"pending"` status but never triggers execution. `retry_publish_task` resets to `"pending"` but also never starts execution.
- **Fix:** After creating tasks in `create_publish_tasks`, schedule `_execute_publish` as a background task:
  ```python
  from fastapi import BackgroundTasks

  @router.post("")
  async def create_publish_tasks(..., background_tasks: BackgroundTasks):
      # ... create tasks ...
      for task in tasks:
          background_tasks.add_task(_execute_publish, task.id, db_url)
      return tasks
  ```
  Similarly in `retry_publish_task`:
  ```python
  background_tasks.add_task(_execute_publish, task_id, str(db.bind.url))
  ```
  Also fix `_execute_publish` to use its own session since background tasks run outside the request lifecycle.
- **Testing:** Create a publish task via API, verify status transitions from `pending` → `running` → `success`/`failed`. Check `publish_logs` table for log entries.

### 1.2 Login Credentials Exposed in URL Query Parameters
- **Files:** `server/app/routes/auth.py:30-31`, `desktop/src/services/api.ts:42`
- **Root Cause:** On server, bare `str` parameters in POST endpoints are interpreted by FastAPI as query parameters. On frontend, `params: { username, password }` sends them in the URL.
- **Fix (Server):** Create a Pydantic model and use it as the request body:
  ```python
  # server/app/schemas/user.py - add:
  class LoginRequest(BaseModel):
      username: str
      password: str

  # server/app/routes/auth.py:
  @router.post("/login")
  def login(request: LoginRequest, db: Session = Depends(get_db)):
      user = db.query(User).filter(User.username == request.username).first()
      if not user or not verify_password(request.password, user.password_hash):
          raise HTTPException(status_code=401, detail="Invalid credentials")
      # ...
  ```
- **Fix (Frontend):** Update `api.ts` login to send JSON body:
  ```typescript
  login(username: string, password: string) {
    return this.client.post<Token>('/auth/login', { username, password })
  }
  ```
- **Testing:** Call `POST /api/auth/login` with JSON body and verify it works. Check server logs to confirm credentials are NOT in URL.

### 1.3 CORS Wildcard with Credentials
- **File:** `server/app/main.py:30-36`
- **Root Cause:** `allow_origins=["*"]` with `allow_credentials=True` violates CORS spec and enables cross-origin attacks.
- **Fix:** Use environment variable for allowed origins:
  ```python
  # server/app/config.py - add:
  CORS_ORIGINS: str = os.getenv("CORS_ORIGINS", "http://localhost:5173")

  # server/app/main.py:
  origins = [o.strip() for o in settings.CORS_ORIGINS.split(",")]
  app.add_middleware(
      CORSMiddleware,
      allow_origins=origins,
      allow_credentials=True,
      allow_methods=["*"],
      allow_headers=["*"],
  )
  ```
- **Testing:** Verify requests from `localhost:5173` succeed. Verify requests from other origins are blocked.

### 1.4 Hardcoded Secret Key Propagated Across 3 Locations
- **Files:** `server/.env:2`, `server/app/config.py:15`, `docker-compose.yml:14`
- **Root Cause:** Weak default `SECRET_KEY=smilex-cas-secret-key-change-in-production` with code fallback means deployment without changing it is trivially exploitable.
- **Fix:**
  1. In `config.py`, remove the hardcoded default and fail if not set:
     ```python
     SECRET_KEY: str = os.getenv("SECRET_KEY")
     if not SECRET_KEY:
         raise ValueError("SECRET_KEY environment variable is required")
     ```
  2. Create `server/.env.example` with `SECRET_KEY=changeme-generate-a-random-key`
  3. Remove `SECRET_KEY` from `docker-compose.yml`, use `env_file` or Docker secrets instead
- **Testing:** Start server without `SECRET_KEY` env var → should fail with clear error. Start with a random key → should work.

### 1.5 Default Admin Password Hardcoded
- **File:** `server/app/database.py:39,46-48`
- **Root Cause:** Password `admin123` is hardcoded and printed to stdout.
- **Fix:** Read from environment variable, fail if not set on first run:
  ```python
  admin_password = os.getenv("ADMIN_DEFAULT_PASSWORD")
  if not admin_password:
      admin_password = "admin123"
      logger.warning("Using default admin password. Set ADMIN_DEFAULT_PASSWORD env var!")
  ```
  Additionally, force password change on first login by adding a `must_change_password` flag to the User model.
- **Testing:** Start with fresh DB → verify admin account created. Verify log warning appears.

## Priority 2: HIGH Severity Issues

### 2.1 Article `metadata` Field Always Returns `None` (Attribute Name Mismatch)
- **Files:** `server/app/models/article.py:22`, `server/app/schemas/article.py:38`
- **Root Cause:** SQLAlchemy model uses `extra_data = Column("metadata", JSON)` but Pydantic schema expects `metadata` attribute. `from_attributes=True` looks for `article.metadata` which doesn't exist as a mapped attribute.
- **Fix:** Either rename the model attribute to match, or add a Pydantic alias:
  ```python
  # Option A (recommended): Rename model attribute
  # server/app/models/article.py:
  metadata = Column(JSON, default=dict)  # use "metadata" as both attr and column name

  # Option B: Use Pydantic alias
  # server/app/schemas/article.py:
  metadata: Optional[Dict[str, Any]] = Field(None, validation_alias="extra_data")
  ```
  Also fix `ArticleUpdate.metadata` update in `routes/articles.py:118-119` — `setattr(article, "metadata", value)` won't work if the attribute is named `extra_data`.
- **Testing:** Create article with metadata, fetch it, verify metadata is returned. Update metadata, verify persistence.

### 2.2 No File Upload Size Limit (Memory Exhaustion / DoS)
- **File:** `server/app/routes/media.py:29,81`
- **Root Cause:** `content = await file.read()` reads entire file into memory without size limit.
- **Fix:**
  ```python
  MAX_UPLOAD_SIZE = 100 * 1024 * 1024  # 100MB

  @router.post("/upload")
  async def upload_file(file: UploadFile, ...):
      content = await file.read(MAX_UPLOAD_SIZE + 1)
      if len(content) > MAX_UPLOAD_SIZE:
          raise HTTPException(status_code=413, detail="File too large")
      # ...
  ```
- **Testing:** Upload a file > 100MB → should get 413 error. Upload normal file → should succeed.

### 2.3 No File Type Validation + Unauthenticated Static File Serving
- **Files:** `server/app/routes/media.py:25,77`, `server/app/main.py:38`
- **Root Cause:** No file extension/MIME type validation. Combined with public static file serving at `/uploads/`, enables stored XSS.
- **Fix:**
  ```python
  ALLOWED_EXTENSIONS = {".jpg", ".jpeg", ".png", ".gif", ".webp", ".mp4", ".mov", ".avi", ".mp3", ".wav", ".pdf"}

  def validate_file_type(filename: str):
      ext = Path(filename).suffix.lower()
      if ext not in ALLOWED_EXTENSIONS:
          raise HTTPException(status_code=400, detail=f"File type {ext} not allowed")

  # Remove public static mount, serve files through authenticated endpoint:
  @router.get("/files/{file_path:path}")
  async def serve_file(file_path: str, user = Depends(get_current_user)):
      full_path = UPLOADS_DIR / file_path
      if not full_path.resolve().is_relative_to(UPLOADS_DIR.resolve()):
          raise HTTPException(status_code=403)
      return FileResponse(full_path)
  ```
- **Testing:** Upload `.html` file → should be rejected. Upload `.jpg` → should succeed. Access `/uploads/` directly → should require auth.

### 2.4 Hardcoded API Base URL / Non-Functional Settings Page
- **Files:** `desktop/src/services/api.ts:16`, `desktop/src/pages/Settings.tsx:8-10`
- **Root Cause:** `baseURL` is hardcoded to `http://localhost:8000/api`. Settings page shows success toast but doesn't persist anything. `electron-store` is a dependency but unused.
- **Fix:**
  ```typescript
  // desktop/src/services/api.ts:
  import Store from 'electron-store' // won't work in renderer - use IPC instead

  // In preload.ts, expose store access:
  contextBridge.exposeInMainWorld('electronAPI', {
    platform: process.platform,
    getApiUrl: () => store.get('apiUrl', 'http://localhost:8000/api'),
    setApiUrl: (url: string) => store.set('apiUrl', url),
  })

  // In api.ts:
  const baseURL = window.electronAPI?.getApiUrl() || 'http://localhost:8000/api'
  ```
  Make Settings page actually call `window.electronAPI.setApiUrl()`.
- **Testing:** Change API URL in Settings → verify persisted across app restarts. Set wrong URL → verify error handling.

### 2.5 Auth State Inconsistency (getCurrentUser.rejected Not Handled)
- **File:** `desktop/src/store/authSlice.ts:59-74`
- **Root Cause:** No `.addCase(getCurrentUser.rejected, ...)` handler. On token expiry, `isAuthenticated` remains `true` causing redirect loops and broken UI.
- **Fix:**
  ```typescript
  builder.addCase(getCurrentUser.rejected, (state) => {
    state.isAuthenticated = false
    state.token = null
    state.user = null
    localStorage.removeItem('token')
  })
  ```
  Also fix the axios interceptor in `api.ts:31-34` to dispatch `logout()` instead of `window.location.href`:
  ```typescript
  // Use a logout callback instead of direct navigation
  if (error.response?.status === 401) {
    localStorage.removeItem('token')
    window.location.href = '/login'
  }
  ```
- **Testing:** Login → wait for token expiry → verify redirect to login (not infinite loop). Login → manually corrupt token → verify clean logout.
